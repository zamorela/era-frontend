## «Очередь генераций» для ERA2

> Реализовать экран **«Очередь генераций»** по макету Figma и со «живым» движком очереди.
> Бэкенда нет — все данные и асинхронность эмулируются на клиенте по спецификации ниже.

---

## 1. Контекст

**ERA2** — агрегатор нейросетей: одна подписка даёт доступ к 90+ моделям (текст, изображения, видео, аудио). Пользователь отправляет задачи на генерацию, они выполняются не мгновенно и обрабатываются **очередью** с ограничением по числу одновременных задач. Нужно показать эту очередь и дать ей управлять.

Этой страницы в продукте **ещё нет** — вы делаете её с нуля.

**Стек проекта:**
- React 19, TypeScript (strict)
- сброщик Vite
- Tailwind CSS
- Стейт-менеджмent — useReducer / Zustand / RTK.
- Иконки lucide-react


### Архитектура — Feature-Sliced Design (FSD)

Проект построен по **FSD**:

```
app → pages → widgets → features → entities → shared
```

- **app/** — инициализация: провайдеры, роутинг, глобальные стили. Трогать почти не нужно.
- **pages/** — тонкая страница-композиция: только собирает виджеты, без бизнес-логики.
- **widgets/** — самостоятельный композитный блок (весь экран очереди = один виджет).
- **features/** — пользовательские сценарии и их состояние (вся логика очереди: автомат, движок, действия).
- **entities/** — бизнес-сущности (задача генерации: типы, данные).
- **shared/** — переиспользуемое и не знающее о бизнесе (ui-kit, `lib`, хуки).

**Сегменты внутри слайса:** `ui/` (компоненты) · `model/` (состояние, типы, редьюсер, селекторы, движок) · `api/` (данные/моки) · `lib/` (утилиты).
**Публичный API слайса — только его `index.ts`.** Импорт между слайсами идёт через `index.ts`, **без deep-import**: `@/features/generation-queue` — да, `@/features/generation-queue/ui/TaskRow` — нет. Алиас `@/` = `src/`.

**Правила:**
- Один компонент — один файл, имя файла = имя компонента.
- Вся бизнес-логика (движок, редьюсер, селекторы) — в `features/generation-queue/model/`, **не в компонентах**. Компоненты — «тупые», получают данные и колбэки.
- Типы домена — только в `entities/generation-task/model/types.ts`, переиспользуются везде.
- Никаких «utils-помоек» в корне: утилита живёт в `lib/` своего слайса.
- У каждого слайса есть `index.ts`; наружу торчит только то, что в нём.
- Импорты — через публичные API слайсов, без `../../..` и deep-import.

## 2. Что нужно сделать

Экран со списком задач генерации, который **живёт во времени**: задачи сами продвигаются по статусам, прогресс растёт, часть падает с ошибкой. Пользователь может фильтровать, сортировать, искать и управлять задачами (отмена / повтор / удаление).


В файле три страницы:
- **`Foundations`** — палитра (Variables), текстовые стили, базовые компоненты (button/card/input/badge/pill) — берите отсюда токены и стили;
- **`Queue`** — экран очереди в 3 брейкпоинтах (`Queue / Desktop · Tablet · Mobile`);
- **`Chat + Status`** — экран чата + плавающий статус-бар в 3 брейкпоинтах (`Chat + Status / Desktop · Tablet · Mobile`) и доска состояний `Status bar · состояния`.

---

## 3. Мок-движок очереди

Реализуйте «движок», который эмулирует обработку. Требования:

1. **Лимит параллелизма** `MAX_CONCURRENT = 2`. Одновременно в статусе `running` не больше двух задач. Освободился слот — берём следующую `queued` (FIFO по `createdAt`).
2. **Прогресс**: у `running`-задачи `progress` растёт тиками (например, каждые ~400–700 мс на случайный шаг). Достиг 100 → `done`.
3. **Случайные сбои**: с вероятностью **~15%** задача в процессе падает в `failed` с понятным текстом ошибки (варьируйте: «Недостаточно кредитов», «Превышено время ожидания», «Модель временно недоступна»).
4. **Скорость зависит от типа**: video/audio «генерируются» заметно дольше, чем text/image (заложите разные длительности).
5. **Корректные таймеры**: при размонтировании/остановке — чистить интервалы; при `cancel` — немедленно прекращать прогресс конкретной задачи (без «дотиков»).
6. Движок не должен мутировать состояние напрямую из нескольких мест бесконтрольно — продумайте единый источник правды.

> Реальные сетевые вызовы делать **не нужно**. Допустимо обернуть тики в `setInterval`/`setTimeout` или эмулировать «стрим» прогресса. Главное — корректность переходов, лимит слотов и чистка ресурсов.

Стартовый датасет — **8–12 задач** в разных статусах (можно захардкодить сид), чтобы при загрузке экран был «живым»: пара running с прогрессом, несколько queued, готовые, минимум одна failed.

## 4. Функциональные требования (UI)

### 4.1 Шапка экрана
- Заголовок «Очередь генераций» + подзаголовок.
- Кнопка **«Очистить готовые»** — удаляет все задачи в статусе `done` (с подтверждением или undo — на ваш вкус, undo = бонус).

### 4.2 Сводка (4 счётчика)
Карточки **В очереди / Идёт / Готово / Ошибка** с актуальными числами. Значения **реактивны** — меняются по мере работы движка.

### 4.3 Тулбар
- **Фильтр по статусу**: чипы `Все · В очереди · Идёт · Готово · Ошибка`. Активный выделен.
- **Сортировка**: минимум «Сначала новые / Сначала старые». (По статусу/прогрессу — бонус.)
- **Поиск** по тексту промпта (debounce). _На desktop-макете показан как сортировка — поиск добавьте логически; разместите рядом._
- (Бонус) фильтр по типу (текст/изображение/видео/аудио).

### 4.4 Список задач
Каждая задача (строка на desktop, карточка на mobile):
- иконка/превью по типу (для image/video — заглушка-плейсхолдер),
- промпт (с обрезкой длинного текста),
- пилюля модели + мета (ETA / длительность / кредиты / позиция в очереди),
- статусный бейдж нужного цвета,
- для `running` — **прогресс-бар** + проценты, обновляются в реалтайме,
- для `failed` — текст ошибки,
- действия по статусу:
  - `running` / `queued` → **Отмена**
  - `failed` / `canceled` → **Повторить**
  - `done` → **Скачать** (заглушка)
  - всегда → меню «…» (как минимум «Удалить»).

### 4.5 Состояния экрана
- **Пустое состояние** (нет задач / нет результатов под фильтром) — отдельный осмысленный вид, не пустой `div`.
- **Загрузка** — скелетоны или спиннер при первичной «загрузке» сида (эмулируйте задержку ~600 мс).
- **Ошибка загрузки** — состояние с кнопкой «Повторить» (эмулируйте возможный сбой инициализации).

### 4.6 Адаптив
- **Desktop (≥1024)** — строки списка, как в макете.
- **Mobile (≤480)** — карточки в стек, статистика 2×2, чипы со скроллом по горизонтали.
- Промежуток (планшет) — не ломается. Точные брейкпоинты — на ваше усмотрение, ориентир — макет.

### 4.7 Персистентность
Состояние очереди сохраняется в `localStorage` и восстанавливается при перезагрузке (running-задачи при восстановлении логично перевести в queued или продолжить — решение опишите).

### 4.8 Глобальный статус-бар генераций (важно)

Кроме страницы очереди нужен **глобальный плавающий индикатор**, который виден поверх любого экрана, пока есть активные генерации, — как «менеджер загрузок» в браузере. В макете он показан поверх экрана чата (фреймы `Chat + Status / Desktop · Tablet · Mobile`), все его состояния — на доске `Status bar · состояния`.

> Сам экран чата верстать **не нужно** — он в макете только чтобы показать, *где* живёт индикатор. Достаточно вывести индикатор поверх любого простого фона/страницы.

**Состояния (зависит от количества активных задач — `running` + `queued`):**
- **нет активных** → индикатор скрыт;
- **1 задача** → компактная карточка: спиннер, тип/модель, мини прогресс-бар + %;
- **несколько** → раскрытый виджет: заголовок «Генерации идут · N активны · X%» (X — усреднённый прогресс), мини-список 2–3 задач с прогрессом и кнопка-ссылка **«Открыть очередь →»**;
- **свёрнутый** (collapsed) — компактная пилюля «N генераций · X%», разворачивается по клику *(опционально, бонус)*.

**Поведение:**
- Клик по виджету / «Открыть очередь» → переход на страницу очереди (`/queue`).
- **Единый источник правды:** индикатор и страница читают один и тот же стор — счётчики и прогресс всегда совпадают. Дублировать состояние нельзя.
- Появление/скрытие — плавно; бар не должен «дёргаться» на каждом тике прогресса.

**Размещение (адаптив, по макету):**
- **Desktop / tablet** — плавающий, прибит **снизу-справа** (отступ ~24px), поверх контента.
- **Mobile** — **полноширинная панель снизу**, над зоной ввода/нижней навигацией, с учётом safe-area.

**По FSD:** индикатор — часть той же фичи `features/generation-queue` (рядом с виджетом очереди), читает тот же контекст; монтируется глобально на уровне `app`/`widgets`. Роутинг до `/queue` — любой простой (react-router или условный рендер; решение опишите в README).

## 5. Дизайн и точность вёрстки

- Свёрстать **по макету**: отступы, радиусы, цвета, типографика. Пиксель-в-пиксель не требуем, но «на глаз похоже» — обязательно.
- Цвета берите из токенов дизайн-системы (`Foundations`): акцент `#E85420`, тёмная тема «warm coal». Статусы: queued — нейтральный, running — оранжевый, done — зелёный, failed — красный, canceled — приглушённый.
- Шрифт **Geist** (моно — Geist Mono для чисел/моделей). Если нет под рукой — Inter как фолбэк, отметьте в README.
- Анимации (рост прогресса, появление/удаление строк, hover) — приветствуются, но без фанатизма.

## 6. Бонусные баллы (по желанию, не обязательны)

- Юнит-тесты на логику движка/редьюсера (переходы, лимит слотов).
- **Optimistic UI** + **Undo** (например, на удаление / «очистить готовые»).
- Виртуализация списка (представьте 1000 задач).
- Drag-to-reorder для `queued` - задач.
- Доступность: фокус, `aria`, управление с клавиатуры, `prefers-reduced-motion`.
- Анимации появления/удаления (framer-motion).
- Светлая тема (в дизайн-системе есть оба набора токенов).
